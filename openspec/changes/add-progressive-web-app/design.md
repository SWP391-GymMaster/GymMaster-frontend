## Context

GymMaster is a Next.js 16.2 App Router frontend with role-aware protected workspaces and an MSW browser worker used when `NEXT_PUBLIC_API_MOCKING=enabled`. The repository currently exposes only a regular icon in root metadata; it has no `manifest.ts`, PWA registration, application service worker, offline fallback, install UX, or PWA dependency. `public/mockServiceWorker.js` is generated by MSW and cannot also serve as the production PWA worker.

Next.js 16 provides native App Router manifest support through `src/app/manifest.ts`. Its current PWA guidance treats Serwist as an optional offline integration that requires webpack configuration. GymMaster needs a conservative, auditable cache boundary more than an offline-first data layer, because its authenticated membership, payment, check-in, workout, and nutrition data must not be stored indiscriminately.

Stakeholders are all four user roles, with mobile members as the most likely installers and staff operations as the most sensitive workflows. Implementation remains frontend-only and must preserve authentication, RBAC, Vietnamese runtime copy, responsive behavior, and the GymMaster OS visual system.

## Goals / Non-Goals

**Goals:**

- Make the production frontend installable with correct GymMaster branding.
- Provide a reliable offline document and connectivity feedback while keeping business mutations online-only.
- Establish an explicit, testable cache allowlist that excludes protected and user-specific traffic.
- Prevent the PWA worker from colliding with the existing root-scoped MSW worker.
- Give users control over installation and worker updates.
- Keep the implementation compatible with the current Next.js standalone deployment.

**Non-Goals:**

- Offline authentication, offline dashboards, offline reads of protected records, background sync, or queued mutations.
- Push notifications, badges, shortcuts, share-target/file-handler APIs, or periodic background work.
- Backend, database, API-contract, RBAC, route-map, or product-scope changes.
- Replacing MSW, changing the package manager, or adding a PWA framework in this change.
- Guaranteeing install prompts on browsers that do not expose the required browser capability.

## Decisions

### 1. Use native Next.js manifest support and a small repository-owned worker

Create `src/app/manifest.ts` using `MetadataRoute.Manifest`, generate GymMaster PNG icons in `public/icons/`, and implement a focused `public/sw.js`. No dependency is added.

This is preferred over Serwist for the approved scope because GymMaster does not need precaching of the entire authenticated application, the cache policy is intentionally narrow, and the current Next.js guidance notes that Serwist integration requires webpack configuration. Avoiding that integration also avoids changing the Next.js 16 build pipeline from its current defaults. Serwist can be reconsidered in a future spec if offline-first features become a product requirement.

### 2. Treat the worker as a public-shell safety layer, not a data cache

The worker uses versioned cache names under a unique `gymmaster-pwa-` prefix. Install pre-caches only the standalone Vietnamese offline document and PWA brand icons. Runtime caching is limited to same-origin immutable `/_next/static/` assets and explicitly named public GymMaster brand assets using cache-first behavior with network population.

Document navigations remain network-first and successful protected HTML/RSC responses are not written to Cache Storage. On network failure, a document navigation receives `public/offline.html`. Requests are bypassed when they are non-GET, cross-origin, under `/api`, contain an RSC/prefetch signal, represent authentication/business operations, or do not match the static allowlist.

This allowlist is intentionally stricter than a generic PWA recipe. It makes cache inspection and privacy tests deterministic and avoids stale role-specific UI or sensitive data surviving logout.

### 3. Coordinate PWA and MSW before either worker starts

Add a client-only PWA lifecycle boundary under the existing root providers. In production with mocking disabled, it registers `/sw.js` after the page becomes interactive. When API mocking is enabled, it finds only registrations whose active/installing/waiting script URL ends with `/sw.js`, unregisters them, deletes only `gymmaster-pwa-*` caches, and then starts MSW. It never unregisters `mockServiceWorker.js` or unrelated registrations.

The PWA suite must run against `npm run build` plus `npm run start`, not `next dev`, because production-only registration is deliberate. This preserves the current development/mock workflow and prevents two root-scoped workers competing for the same clients.

### 4. Keep install and update UX contextual and user-controlled

A reusable client component captures `beforeinstallprompt` and detects standalone/iOS state. The install action is integrated into the existing Admin, Staff, PT, and Member profile surfaces, where installation is discoverable without interrupting operational workflows. iOS receives manual Vietnamese Share/Add to Home Screen guidance. Dismissal is stored locally for the current PWA release and the action is hidden in standalone mode.

The lifecycle component listens for a waiting worker. It uses the existing Sonner infrastructure to offer a persistent Vietnamese update action. The worker waits by default; only explicit acceptance sends `SKIP_WAITING`. After `controllerchange`, the client reloads exactly once. Deferral never auto-reloads or discards a form.

### 5. Use a static, dependency-free offline document

Use `public/offline.html` with self-contained semantic HTML and minimal inline GymMaster OS styling. This avoids depending on React, protected layouts, server rendering, or build-hashed chunks precisely when the network is unavailable. It includes Vietnamese status text, a retry button, keyboard-visible focus, and no member-specific content.

A small global connectivity notice reacts to browser `online`/`offline` events for already-open pages. It is informational only; existing TanStack Query/mutation error handling remains authoritative and no background queue is introduced.

The repository's pre-existing `useOfflineSync` hook is not a real sync implementation: it stores actions in `gymmaster-offline-queue`, waits locally, and displays a success toast without calling an API. Remove that hook from Water Tracker, delete the unused hook, clear the legacy key during PWA lifecycle startup, and label water history as device-local. Water logging may continue to use its dedicated local storage because it is not represented as a confirmed backend mutation.

### 6. Scope response headers to the worker

Extend `next.config.ts` only for `/sw.js`: `Content-Type: application/javascript; charset=utf-8`, `Cache-Control: no-cache, no-store, must-revalidate`, `Content-Security-Policy: default-src 'self'; script-src 'self'`, and `X-Content-Type-Options: nosniff`. Broad global security-header changes are excluded to avoid an unrelated behavioral change.

### 7. Verify behavior at contract and production-browser levels

Add focused tests for manifest fields, install/standalone/iOS states, update lifecycle, connectivity notice, request classification, cache version cleanup, and MSW/PWA selection. Add a Playwright PWA project or focused spec that runs against a production server to verify manifest/icon/worker headers, worker control, offline navigation fallback, update affordance where deterministic, and Cache Storage exclusions.

Manual acceptance uses Chromium Application tooling (and Lighthouse PWA checks when available) plus an iOS Add to Home Screen checklist. Lighthouse is a verification aid, not a new repository dependency or the sole acceptance gate.

## Risks / Trade-offs

- **[Manual worker logic can drift as routes evolve]** → Centralize request classification, keep a strict allowlist, cover it with tests, and require future cache expansion through an approved spec.
- **[A stale production worker can interfere with localhost MSW]** → Perform targeted PWA registration/cache cleanup before MSW startup and test the transition explicitly.
- **[Offline fallback is less capable than an offline-first app]** → State the limitation honestly; never cache protected data or queue writes. Revisit only through a separate capability proposal.
- **[Install events differ by browser and cannot be forced]** → Gate UI on capability/standalone state and provide manual iOS guidance.
- **[A bad worker can persist after rollback]** → Keep `/sw.js` uncacheable and maintain a rollback worker that deletes `gymmaster-pwa-*`, unregisters itself, and reloads clients once.
- **[Cached static assets increase storage]** → Limit by path/type, version caches, delete older versions during activation, and avoid runtime image caches outside approved brand assets.
- **[Production-only behavior is missed by normal dev tests]** → Add a production-build Playwright gate and record manual browser inspection in the change checklist.

## Migration Plan

1. Add manifest/icons and validate every referenced asset before worker registration is enabled.
2. Add the offline document, worker request classifier, response headers, and unit tests.
3. Add the lifecycle coordinator with explicit MSW exclusion/cleanup and update handling.
4. Add profile install actions and connectivity feedback with accessibility tests.
5. Remove the simulated Water Tracker queue, clear its legacy storage key, and verify that no synthetic synchronization success remains.
6. Run typecheck, lint, unit tests, production build, production-mode PWA Playwright tests, and manual browser/iOS checklists.
7. Deploy normally over HTTPS and verify manifest, scope, worker headers, cache contents, online mutations, and one update cycle.

Rollback removes the registration UI/code and deploys a temporary cleanup `/sw.js` that activates immediately, deletes only `gymmaster-pwa-*` caches, unregisters itself, and refreshes controlled clients once. The cleanup worker remains served with no-cache headers until active-client verification shows the old worker population has cleared.

## Open Questions

The following decisions are resolved only when the owner approves this design:

1. Approve the dependency-free native worker instead of adding Serwist and webpack configuration.
2. Approve install actions on the four existing role profile surfaces, with no automatic global install modal.
3. Approve the privacy-first boundary: offline fallback/static assets only; no protected data caching, background sync, or offline mutations.
